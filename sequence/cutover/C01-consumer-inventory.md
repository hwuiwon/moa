# Step C01 — Consumer inventory and classification

_Audit every site that imports or uses `moa_memory::*`. Produce a single markdown table at `docs/migrations/moa-memory-inventory.md` classifying each consumer. The user reviews and approves the table before C02 starts._

## 1 What this step is about

Before migrating consumers, we need to know what they're doing. This prompt produces an inventory: every import, every call site, classified by the kind of migration it needs. The output is a markdown doc the user reviews. Decisions on `WIKI_RETIRE` and `NEEDS_DESIGN` rows are recorded in the doc itself, not made implicitly.

## 2 Files to read

- C00 (cutover overview)
- `crates/moa-memory/src/lib.rs` (the legacy crate's surface area)
- `crates/moa-core/src/traits.rs` (the `MemoryStore` trait definition)
- `crates/moa-core/src/types/` (wiki types: `WikiPage`, `MemoryPath`, `PageType`, etc.)

## 3 Goal

`docs/migrations/moa-memory-inventory.md` exists with one row per consumer site, categorized into the buckets defined in C00. The doc has a "Decisions" section the user fills in before C02. Build is unchanged (no code touched in this step).

## 4 Rules

- **Audit, don't refactor.** This step writes documentation only. No source files change.
- **Be exhaustive.** Every `use moa_memory::` and `use moa_core::{... MemoryStore ...}` site gets a row.
- **Group at the call-site level**, not the file level. One file may have multiple distinct consumer patterns.
- **Honest classification.** If you're not sure whether a consumer is `RETRIEVAL_ROUTE` vs `INGEST_ROUTE`, mark it `NEEDS_DESIGN` and explain in the notes column.

## 5 Tasks

### 5a Run the audit

```sh
# Direct imports of the legacy crate
rg -l "use moa_memory(::|;|\\s)" crates/ --type rust > /tmp/c01-direct.txt

# Imports of the trait or wiki types from moa-core
rg -l "use moa_core::\\{[^}]*MemoryStore" crates/ --type rust > /tmp/c01-trait.txt
rg -l "use moa_core::\\{[^}]*WikiPage" crates/ --type rust > /tmp/c01-wikipage.txt
rg -l "use moa_core::\\{[^}]*MemoryPath" crates/ --type rust > /tmp/c01-memorypath.txt
rg -l "use moa_core::\\{[^}]*PageType" crates/ --type rust > /tmp/c01-pagetype.txt
rg -l "use moa_core::\\{[^}]*IngestReport" crates/ --type rust > /tmp/c01-ingestreport.txt

# Cargo.toml dependencies on moa-memory
rg "moa-memory\\s*=" crates/ --type toml > /tmp/c01-cargo.txt

# FileMemoryStore call sites
rg "FileMemoryStore" crates/ --type rust > /tmp/c01-fms.txt

# Method-level call sites for the trait methods
for method in search read_page write_page delete_page list_pages get_index ingest_source rebuild_search_index search_with_mode; do
    rg "\\.$method\\s*\\(" crates/ --type rust > /tmp/c01-call-$method.txt
done

# Aggregate
cat /tmp/c01-*.txt | sort -u > /tmp/c01-files.txt
wc -l /tmp/c01-files.txt
```

### 5b Read each implicated file

For each file in `/tmp/c01-files.txt`, open it and identify each distinct consumer pattern. Examples of distinct patterns inside a single file:

- One function uses `MemoryStore::search` (read).
- Another function uses `FileMemoryStore::ingest_source` (write).
- A test imports `WikiPage` for fixture construction (test-only).

Each gets its own row in the inventory.

### 5c Classify

Use these buckets (defined in C00):

- `TRIVIAL_DELETE` — dead code or test fixture with no semantic value to preserve.
- `SIMPLE_GRAPH` — small site, swap directly to `GraphStore::*` calls (e.g., a debug helper that just looks up by uid).
- `INGEST_ROUTE` — write path; should route through `IngestionVO::ingest_turn` (slow) or `fast_remember` (fast).
- `RETRIEVAL_ROUTE` — read path; should route through the hybrid retriever in `moa-brain`.
- `WIKI_RETIRE` — feature has no graph equivalent. User decides: retire the feature or redesign it. Note the proposed retire/redesign plan in the notes column.
- `NEEDS_DESIGN` — semantics ambiguous; user makes a call before C02–C04 starts.

### 5d Write the inventory

Create `docs/migrations/moa-memory-inventory.md`:

```markdown

| # | Crate / file:line | Pattern | Bucket | Target API | Notes |
|---|-------------------|---------|--------|------------|-------|
| 1 | `crates/moa-cli/src/main.rs:25` | `use moa_memory::FileMemoryStore` (CLI imports legacy store) | (depends on row 2/3/4) | n/a (import only) | Replaced when call sites are migrated. |

## Decisions for WIKI_RETIRE rows

For each `WIKI_RETIRE` row above, the user records the chosen disposition here:

- **Row 5 — `moa memory rebuild-index`**: [retire | redesign as `<spec>`]
- ...

## Decisions for NEEDS_DESIGN rows

- **Row N — [description]**: [decision + reasoning]
- ...

## Open questions

- (any ambiguities the agent flagged for the user)
```

### 5e Sanity-check totals

```sh
# expected: total sites in inventory >= total files in /tmp/c01-files.txt
# (one file may produce multiple sites)
wc -l /tmp/c01-files.txt
grep -c "^|.*|.*|.*|.*|.*|.*|" docs/migrations/moa-memory-inventory.md
```

The second number should be ≥ the first.

### 5f Stop and wait for review

This is a hard stop. Do not proceed to C02. The user reviews `docs/migrations/moa-memory-inventory.md`, fills in the Decisions sections, and only then runs C02.

## 6 Deliverables

- `docs/migrations/moa-memory-inventory.md` (~150–400 lines depending on consumer count).
- No code changes.

## 7 Acceptance criteria

1. The inventory file exists at the path above.
2. Every file flagged by the audit greps is represented by at least one row.
3. Every row has a non-empty Bucket and Target API column (Target API may be "n/a" for import-only rows).
4. Summary counts at the top of the file are accurate (cross-check by grep).
5. Build is unchanged: `cargo build --workspace` clean, `cargo test --workspace` green.
6. The "Decisions" sections exist (may be empty placeholders for the user to fill in).

## 8 Tests

```sh
test -f docs/migrations/moa-memory-inventory.md
git diff --stat HEAD~1 HEAD | grep -v 'docs/migrations'   # only docs changed
cargo build --workspace
cargo test --workspace
```

## 9 Cleanup

- Delete `/tmp/c01-*.txt` audit artifacts after the inventory is committed.

## 10 What's next
